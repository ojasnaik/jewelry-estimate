# jewelry-estimate

An AI-powered jewelry valuation web application. Users upload photos and enter details about a piece of jewelry — metal type, karat, weight, gemstone, and condition — and receive an estimated resale market value range generated by an LLM.

---

## Features

- **Email authentication** — sign up, sign in, email confirmation via Supabase Auth
- **Jewelry intake form** — metal type, karat (dynamic options per metal), weight, gemstone, condition, optional photo upload
- **LLM valuation** — Groq API (llama-3.3-70b-versatile) estimates a USD resale range with appraiser reasoning
- **Real-time status polling** — pending → processing → complete, with auto-refresh on completion
- **Valuation history dashboard** — table of past valuations with status badges and estimated ranges
- **Signed image URLs** — photos served via short-lived Supabase Storage signed URLs, never exposed directly
- **Error tracking** — Sentry captures exceptions with user context, never logging sensitive data
- **Route protection** — proxy (middleware) redirects unauthenticated users away from `/dashboard`, authenticated users away from `/auth`

---

## Stack

| Layer | Technology |
|---|---|
| Framework | Next.js 16 App Router (TypeScript strict) |
| Auth & Database | Supabase (Auth, PostgreSQL, Storage) |
| LLM | Groq API — llama-3.3-70b-versatile |
| Styling | Tailwind CSS v4 |
| Error tracking | Sentry |
| Hosting | Vercel |


## Database Schema

```sql
create table valuations (
  id              uuid primary key default gen_random_uuid(),
  user_id         uuid references auth.users not null,
  status          text not null default 'pending',  -- pending | processing | complete | error
  metal_type      text,
  karat           text,
  weight_grams    numeric,
  gemstone_type   text,
  gemstone_carat  numeric,
  condition       text check (condition in ('excellent','good','fair','poor')),
  image_path      text,
  estimated_low   numeric,
  estimated_high  numeric,
  llm_reasoning   text,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
```

Row Level Security is enabled. Users can only read and write their own rows.

### 1 — Supabase client factories

> Create the Supabase client factory files.
> - `src/types/supabase.ts` — placeholder `Database = any` type
> - `src/lib/supabase/client.ts` — browser singleton using `createBrowserClient` from `@supabase/ssr`, typed with Database generic
> - `src/lib/supabase/server.ts` — server client using `createServerClient` with Next.js `cookies()` from `next/headers`
>
> Use the official `@supabase/ssr` pattern. Do NOT use `@supabase/auth-helpers-nextjs`. Do NOT import `SUPABASE_SERVICE_ROLE_KEY` in the client file.

**Claude:** Created three files. The server client is `async` because `cookies()` is async in Next.js 15+. The `setAll` in server.ts silently swallows errors when called from a Server Component (read-only cookies) — middleware handles session refresh instead.

### 2 — Proxy / Middleware

> Create `src/proxy.ts` (Next.js 16 middleware) that:
> - Refreshes the session on every request using the Supabase SSR cookie pattern
> - Redirects unauthenticated users accessing `/dashboard` to `/auth/login`
> - Redirects authenticated users accessing `/auth/*` to `/dashboard`
> - Uses `getUser()` not `getSession()`
> - Skips static files via matcher config

**Claude:** Built the client inline using `NextRequest`/`NextResponse` directly — the `cookies()` helper from `next/headers` is not available in middleware. The function was later renamed from `middleware` to `proxy` when Next.js 16 changed the convention.

### 6 — API routes

> Create two API routes:
>
> `POST /api/valuations` — verify session with `getUser()`, validate required fields, insert row with `status='pending'`, fire-and-forget POST to `/api/valuations/process` with `x-process-secret` header, return `{ id, status }` with 201.
>
> `POST /api/valuations/process` — check `x-process-secret` header first, fetch valuation using service role client (bypasses RLS), set status to `'processing'`, call Groq API (`llama-3.3-70b-versatile`, max_tokens 400, temperature 0.1), strip markdown fences before `JSON.parse()`, update row with estimates and `status='complete'`. On any error: set `status='error'`, call `Sentry.captureException` with `{ extra: { valuationId } }`.

### 13 — `requireAuth` utility

> Create `src/lib/auth/requireAuth.ts` — centralised session verification for all API routes. Accept the request, call `getUser()`, return a discriminated union of `{ user, supabase, response: null }` on success or `{ response: NextResponse(401) }` on failure. Update all user-facing API routes to use it.

**Claude:** Returned the `supabase` client alongside the user so routes don't create a second client. The discriminated union means TypeScript enforces the `if (auth.response) return auth.response` guard — forgetting it causes a type error when accessing `auth.user`. The `process/route.ts` route was intentionally excluded (server-to-server, guarded by `x-process-secret` instead).

---

### 14 — Stuck "estimating" on Vercel & poll timeout

> The valuation is stuck on the estimating spinner in production. Where to look, and add a poll timeout.

**Claude:** Added structured logging and Sentry breadcrumbs across the processing pipeline, awaited the previously fire-and-forget dispatch (un-awaited fetches are abandoned when a serverless function returns), and added a 60-second client-side poll timeout in `ValuationPending` that stops polling and marks the row `error` via a new `POST /api/valuations/[id]/timeout` route.

---

### 15 — Root cause: self-fetch blocked by Vercel Deployment Protection

> The dispatch still fails — trace shows `POST /api/valuations/process` returning 401 in ~58 ms. Also, timed-out rows show "estimating" again on reopen.

**Claude:** Diagnosed the 401 as **Vercel Deployment Protection** blocking the server-to-server self-fetch (no `_vercel_jwt` cookie on internal requests) — the request never reached the app, which explained the impossibly fast 58 ms and the 401 (the route itself only ever returns 403). Eliminated the self-fetch entirely: extracted the logic into `src/lib/valuations/processValuation.ts` and ran it via Next.js `after()` inside the create route (`maxDuration = 60`), removing base-URL, cookie, protection, and proxy concerns in one move. Slimmed `process/route.ts` to a thin secret-guarded retry wrapper. Fixed the persistence bug with a server-side staleness check on the result page — rows pending/processing past 2 minutes are marked `error` on load, so they self-heal even if the client navigated away before the in-page timeout fired.
