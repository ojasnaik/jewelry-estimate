# jewelry-estimate

An AI-powered jewelry valuation web application. Users upload photos and enter details about a piece of jewelry ‚Äî metal type, karat, weight, gemstone, and condition ‚Äî and receive an estimated resale market value range generated by an LLM.

---

## Features

- **Email authentication** ‚Äî sign up, sign in, email confirmation via Supabase Auth
- **Jewelry intake form** ‚Äî metal type, karat (dynamic options per metal), weight, gemstone, condition, optional photo upload
- **LLM valuation** ‚Äî Groq API (llama-3.3-70b-versatile) estimates a USD resale range with appraiser reasoning
- **Real-time status polling** ‚Äî pending ‚Üí processing ‚Üí complete, with auto-refresh on completion
- **Valuation history dashboard** ‚Äî table of past valuations with status badges and estimated ranges
- **Signed image URLs** ‚Äî photos served via short-lived Supabase Storage signed URLs, never exposed directly
- **Error tracking** ‚Äî Sentry captures exceptions with user context, never logging sensitive data
- **Route protection** ‚Äî proxy (middleware) redirects unauthenticated users away from `/dashboard`, authenticated users away from `/auth`

## Database Schema

```sql
create table valuations (
  id              uuid primary key default gen_random_uuid(),
  user_id         uuid references auth.users not null,
  status          text not null default 'pending',  -- pending | processing | complete | error
  metal_type      text,
  karat           text,
  weight_grams    numeric,
  gemstone_type   text,
  gemstone_carat  numeric,
  condition       text check (condition in ('excellent','good','fair','poor')),
  image_path      text,
  estimated_low   numeric,
  estimated_high  numeric,
  llm_reasoning   text,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
```

Row Level Security is enabled. Users can only read and write their own rows.

## Key Architecture Decisions

**`requireAuth` utility** ‚Äî all user-facing API routes call `requireAuth(request)` at the top of the handler. It returns the verified `user` and `supabase` client together (one client per request), using a discriminated union so TypeScript enforces the early-return guard at the call site.

**Fire-and-forget processing** ‚Äî `POST /api/valuations` inserts the row and immediately returns `{ id, status: 'pending' }`. It fires a non-awaited `fetch` to `/api/valuations/process` with an `x-process-secret` header. The processing route calls Groq and updates the DB asynchronously, which allows the UI to navigate to the result page immediately and poll for completion.

**`getUser()` everywhere** ‚Äî `getSession()` is never used. `getUser()` re-validates the JWT with Supabase servers on every call, preventing spoofed cookie attacks.

**`proxy.ts`** ‚Äî Next.js 16 renamed `middleware.ts` to `proxy.ts` and the exported function from `middleware` to `proxy`. The file refreshes the session on every request and enforces route-level auth redirects.

**`force-dynamic` on the result page** ‚Äî the valuation detail page (`/dashboard/valuations/[id]`) exports `dynamic = 'force-dynamic'` so that `router.refresh()` called by `ValuationPending` always triggers a fresh DB read rather than hitting a cached render.

### 13 ‚Äî `requireAuth` utility

> Create `src/lib/auth/requireAuth.ts` ‚Äî centralised session verification for all API routes. Accept the request, call `getUser()`, return a discriminated union of `{ user, supabase, response: null }` on success or `{ response: NextResponse(401) }` on failure. Update all user-facing API routes to use it.

**Claude:** Returned the `supabase` client alongside the user so routes don't create a second client. The discriminated union means TypeScript enforces the `if (auth.response) return auth.response` guard ‚Äî forgetting it causes a type error when accessing `auth.user`. The `process/route.ts` route was intentionally excluded (server-to-server, guarded by `x-process-secret` instead).

### 14 ‚Äî Stuck "estimating" on Vercel & poll timeout

> The valuation is stuck on the estimating spinner in production. Where to look, and add a poll timeout.

**Claude:** Added structured logging and Sentry breadcrumbs across the processing pipeline, awaited the previously fire-and-forget dispatch (un-awaited fetches are abandoned when a serverless function returns), and added a 60-second client-side poll timeout in `ValuationPending` that stops polling and marks the row `error` via a new `POST /api/valuations/[id]/timeout` route.

### 15 ‚Äî Root cause: self-fetch blocked by Vercel Deployment Protection

> The dispatch still fails ‚Äî trace shows `POST /api/valuations/process` returning 401 in ~58 ms. Also, timed-out rows show "estimating" again on reopen.

**Claude:** Diagnosed the 401 as **Vercel Deployment Protection** blocking the server-to-server self-fetch (no `_vercel_jwt` cookie on internal requests) ‚Äî the request never reached the app, which explained the impossibly fast 58 ms and the 401 (the route itself only ever returns 403). Eliminated the self-fetch entirely: extracted the logic into `src/lib/valuations/processValuation.ts` and ran it via Next.js `after()` inside the create route (`maxDuration = 60`), removing base-URL, cookie, protection, and proxy concerns in one move. Slimmed `process/route.ts` to a thin secret-guarded retry wrapper. Fixed the persistence bug with a server-side staleness check on the result page ‚Äî rows pending/processing past 2 minutes are marked `error` on load, so they self-heal even if the client navigated away before the in-page timeout fired.
